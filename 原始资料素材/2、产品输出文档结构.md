一、整体前提（先锁死）
	•	主干结构：A → B → C
	•	新增能力：R（审视/审计）作为冻结前的强制产物
	•	当前规范范围：产品需求 AI 规范
	•	本轮目标：让每一份主干文档“为什么存在、填什么、解决什么问题”都清楚

⸻

二、A 类｜现状与基线文档（As-Is）

A0｜产品基础与范围说明

内容结构
	1.	产品背景与定位
	2.	目标用户定义
	3.	核心使用场景
	4.	当前版本范围与边界
	5.	明确不覆盖的内容

填写说明
	•	不写规划、不写愿景
	•	只描述“现在这个产品是什么样”
	•	边界要写清楚（哪些能力没有、哪些不支持）

目的 / 承载
	•	给 AI 和人一个统一的“现状语境”
	•	防止后续规划“假设一个不存在的产品”
	•	作为所有 B 规划的前置事实引用源

⸻

A1｜已上线功能与流程清单

内容结构
	1.	功能列表（按模块）
	2.	核心用户路径
	3.	关键业务流程节点
	4.	功能之间的依赖关系

填写说明
	•	功能是“客观存在的”，不是“设计过的”
	•	用户路径用真实使用顺序，不用理想流程
	•	不评价好坏，只陈述事实

目的 / 承载
	•	让审视与规划基于真实系统
	•	防止 AI 反复“重建已有能力”
	•	为 R0 / R1 提供审视对象

⸻

A2｜存量反馈与数据输入汇总（可选但强烈建议）

内容结构
	1.	用户反馈摘要
	2.	数据异常或指标变化
	3.	内部问题/投诉
	4.	已知未解决事项

填写说明
	•	不做结论、不做方案
	•	可以是原始反馈的整理
	•	标注来源即可

目的 / 承载
	•	为 B 规划提供动因素材
	•	防止规划“拍脑袋”
	•	为 R 审视提供“现实校验”

⸻

三、R 类｜审视 / 审计文档（新增，但不改变主干）

R0｜基线审视报告（A 阶段，初始化用）

内容结构
	1.	审视范围说明
	2.	实际用户路径审视
	3.	主要问题与机会
	4.	风险与隐患
	5.	总体判断结论

填写说明
	•	必须“从头到尾走一遍系统”
	•	问题基于事实，不基于偏好
	•	结论是“是否适合作为稳定基线”

目的 / 承载
	•	给存量系统建立一个起点基线
	•	为第一次 B 规划提供“共识事实”
	•	防止一上来就大改而不知问题在哪

⸻

R1｜规划审视报告（B3 冻结前强制）

内容结构
	1.	审视对象（对应 B 规划）
	2.	目标与场景一致性
	3.	是否基于现状（A）
	4.	主要遗漏与风险
	5.	是否允许冻结结论

填写说明
	•	不提新需求，只审视已有规划
	•	重点看“有没有想清楚”“有没有越界”
	•	结论必须明确（通过 / 需补充 / 否决）

目的 / 承载
	•	阻断“未经思考就冻结”的规划
	•	给 B3 一个被审查过的背书
	•	为 C 阶段减少反复

⸻

四、B 类｜规划与冻结文档（To-Be）

B1｜需求规划草案（AI 对话核心产物）

内容结构
	1.	规划目标
	2.	使用场景
	3.	目标用户
	4.	核心需求描述
	5.	明确不做的内容

填写说明
	•	用“产品语言”，不用技术语言
	•	不拆太细，但要完整
	•	“不做什么”必须写

目的 / 承载
	•	把模糊想法收敛成可讨论对象
	•	作为 B2 拆解的输入
	•	是 AI 深度思考的主战场

⸻

B2｜规划拆解与范围界定

内容结构
	1.	需求项列表
	2.	优先级排序
	3.	范围界定（包含 / 不包含）
	4.	前置条件与依赖

填写说明
	•	开始有“版本意识”，但还不是版本需求
	•	拆解粒度以“是否可独立评估”为准
	•	不引入实现方案

目的 / 承载
	•	为冻结做准备
	•	防止 B3 成为“模糊大饼”
	•	为 R1 提供审视素材

⸻

B3｜规划冻结归档文档

内容结构
	1.	冻结范围说明
	2.	最终规划描述
	3.	已解决与未解决问题
	4.	审视结论引用（R1）
	5.	下游进入 C 的说明

填写说明
	•	冻结后内容不可随意改
	•	未解决的问题必须显性记录
	•	明确“下一步进入版本需求”

目的 / 承载
	•	形成稳定、可传递的规划结果
	•	作为 C 节点的唯一合法输入
	•	防止规划阶段无限拉扯

⸻

五、C 类｜版本需求节点（Version）

C0｜版本范围声明

内容结构
	1.	版本目标
	2.	版本包含范围
	3.	不包含内容
	4.	版本约束

填写说明
	•	是 B3 的“版本化翻译”
	•	不再讨论是否合理
	•	只讨论这一版做什么

目的 / 承载
	•	把规划转成版本承诺
	•	为后续任何执行规范提供清晰边界

⸻

C1｜版本需求清单

内容结构
	1.	需求列表（版本内）
	2.	每条需求的目的说明
	3.	关键业务规则

填写说明
	•	粒度已接近可执行
	•	仍然不写技术实现
	•	需求必须可被验证

目的 / 承载
	•	成为后续规范（AODW 等）的直接输入
	•	防止“规划与版本脱节”

⸻

C2｜版本变更与差异说明（如有）

内容结构
	1.	变更原因
	2.	变更内容
	3.	对版本目标的影响

填写说明
	•	只记录“已发生的变更”
	•	不做评判，只做记录

目的 / 承载
	•	保证版本决策可追溯
	•	防止版本失控

⸻

C3｜版本冻结归档（如你们体系中存在）

内容结构
	1.	最终版本范围
	2.	冻结声明
	3.	审视结论引用（R2，如有）

目的 / 承载
	•	标志“产品需求阶段完成”
	•	为后续任何执行体系提供稳定输入
