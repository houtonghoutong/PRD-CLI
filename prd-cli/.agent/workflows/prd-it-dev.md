# IT-DEV 编写工作流

本工作流旨在协助将业务需求（BIZ）转化为可执行的开发规格（DEV）。

> ⚠️ **核心原则：从 BIZ 提取，不重复不编造**

---

## 🚨 DEV 文档定位

| BIZ 文档 | DEV 文档 |
|---------|---------|
| What（做什么）| How（怎么实现）|
| 用户故事、验收标准 | RT 拆解、技术约束 |
| 业务规则 | 影响范围、涉及模块 |

**DEV 的核心价值是 RT 拆解，不是重复 BIZ 内容！**

---

## 1. 从 BIZ 提取已知信息

DEV 文档应该**引用**而不是**重复**BIZ 中已有的内容：

```markdown
## 1. 需求概述（引用 BIZ）

> 📋 详细验收标准见 [IT-xxx-BIZ.md](./IT-xxx-BIZ.md)

**需求描述**: [从 BIZ 用户故事提取，一句话总结]
**验收标准数**: X 个
**涉及业务规则**: X 个
```

**不要把 BIZ 中已有的状态流转、通知渠道等内容再列一遍！**

---

## 2. DEV 核心内容

### 2.1 必填项（从 BIZ 提取 + 技术判断）

| 字段 | 说明 | 谁填 |
|-----|------|-----|
| 需求描述 | 一句话总结（从 BIZ 提取）| AI 可填 |
| 需求类型 | Feature / Bug / Enhancement | AI 可填 |
| 预估影响范围 | 哪些功能/页面受影响 | AI 可推断 |
| 涉及模块 | 前端/后端/哪些服务 | AI 可推断 |
| 是否涉及 UI | 是否有界面改动 | AI 可判断 |

### 2.2 待确认项（需技术负责人确认）

| 字段 | 说明 |
|-----|------|
| 技术约束 | 必须使用某技术/不能改某代码 |
| 关联需求 | 与其他 RT 的依赖关系 |
| 风险评估 | 低 / 中 / 高 |

### 2.3 开发要点（供技术负责人参考）

DEV 文档只需列出**开发要点提示**，具体 RT 拆解由技术负责人决定：

```markdown
## 3. 开发要点

> ⚠️ 以下仅为开发要点提示，具体 RT 拆解由技术负责人决定

### 涉及的验收标准
- AC-001: xxx
- AC-002: xxx
- ...

### 可能的开发方向
- 前端: xxx页面、xxx组件
- 后端: xxx服务
- 其他: xxx
```

**AI 不应该输出具体的 RT 编号和工时评估！**

---

## 3. AI 禁止事项

| 禁止 | 原因 |
|------|------|
| ❌ 重复 BIZ 中已有的业务规则 | BIZ 是唯一真理来源 |
| ❌ 编造 API 设计 | 接口由技术负责人设计 |
| ❌ 编造数据库设计 | 表结构由开发团队设计 |
| ❌ 大量留空表格 | DEV 不是问卷调查 |

---

## 4. DEV 文档模板（正确版）

```markdown
# IT-xxx-DEV: [名称]

**关联BIZ**: [IT-xxx-BIZ.md](./IT-xxx-BIZ.md)
**状态**: 待确认

---

## 1. 需求概述

> 📋 详细验收标准见 BIZ 文档

**需求描述**: [从 BIZ 一句话提取]
**需求类型**: Feature
**验收标准数**: X 个

---

## 2. 影响范围

| 项目 | 内容 |
|-----|------|
| 涉及模块 | 前端: xxx / 后端: xxx |
| 是否涉及 UI | 是 / 否 |
| 影响页面 | xxx |
| 外部依赖 | xxx |

---

## 3. 开发要点

> ⚠️ 以下仅为开发要点提示，具体 RT 拆解由技术负责人决定

### 涉及的验收标准
- AC-001: xxx
- AC-002: xxx

### 可能的开发方向
- 前端: xxx
- 后端: xxx

---

## 4. 待确认问题

1. [具体的技术决策问题]
2. [具体的依赖问题]

---

## 技术负责人确认

- [ ] 影响范围确认
- [ ] 开发要点已评审
- [ ] RT 已拆解（记录在任务系统中）
- [ ] 可以开始开发
```

---

## 5. 结束

提醒用户：
> "DEV 文档提供开发要点，具体 RT 拆解由技术负责人在任务系统中完成。"
