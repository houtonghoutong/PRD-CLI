# P0_项目基本信息填写工作流

## 📌 P0 文档的定位

P0_项目基本信息.md 是整个 PRD 体系的**起点文档**，在创建其他任何文档之前必须先完善。

### P0 的作用

1. **确认项目存在的合理性**：明确项目是否应该存在
2. **建立共识基础**：为所有后续工作建立共同的理解基础
3. **识别干系人**：明确项目相关的关键人员
4. **记录约束条件**：明确时间、资源、依赖等限制

### P0 与 A 类文档的关系

- **P0**: 回答"项目是什么、为什么要做"
- **A0**: 回答"产品现在是什么样的"
- **A1**: 回答"产品目前有哪些功能"
- **A2**: 回答"用户有哪些反馈和问题"

**执行顺序**: P0 → A0 → A1 → A2 → R0

---

## 🚦 填写时机

### 执行 `prd init <projectName>` 后

1. 系统会自动创建 P0_项目基本信息.md 模板
2. **立即打开并完善 P0 文档**
3. 完成 P0 填写后，再执行 `prd baseline create A0`

### ⚠️ 重要提醒

**禁止跳过 P0 直接创建 A0！**

如果没有完善 P0：
- AI 无法理解项目背景和目标
- 后续文档可能偏离项目初衷
- 干系人不清晰，决策链混乱
- 约束条件不明确，规划可能不切实际

## 📝 P0 填写流程

### 第 1 步：项目基本信息

**AI 引导问题**：

```
AI: "让我们一起完善 P0 项目基本信息。首先明确项目定位：

     1. 项目全称是什么？
     2. 用一句话描述这个项目是什么？
     3. 这个项目属于哪个产品线？
     4. 项目级别是什么？（战略级/业务级/支撑级）"
```

**PM 需要回答的内容**：
- 项目全称（明确、无歧义）
- 项目简述（一句话说清楚）
- 所属产品线（在整体业务中的位置）
- 项目级别（决定资源优先级）

### 第 2 步：项目目标

**AI 引导问题**：

```
AI: "现在明确项目目标。请回答：

     1. 这个项目要解决的主要问题是什么？（最多 3 个核心问题）
     2. 如何判断项目成功？请给出可衡量的指标。
     3. 为什么现在做这个项目？（时机/背景/触发因素）
     4. 如果不做会怎样？（说明紧迫性）"
```

**PM 需要提供**：
- **核心问题**（不超过 3 个，优先级排序）
- **成功标准**（必须是可测量的指标）
- **时机背景**（为什么是现在）
- **紧迫性**（不做的后果）

**示例**：

❌ **错误的目标**：
- "提升用户体验"（无法衡量）
- "优化系统性能"（太宽泛）
- "增加功能"（手段而非目标）

✅ **正确的目标**：
- "将用户创建成功率从 60% 提升至 90%"
- "将页面加载时间从 3 秒降至 1 秒以内"
- "减少客服投诉量从 100 次/周降至 10 次/周"

### 第 3 步：干系人

**AI 引导问题**：

```
AI: "明确项目干系人：

     1. PM（产品负责人）是谁？
     2. 技术负责人是谁？
     3. 业务方是哪个部门？联系人是谁？
     4. 还有哪些团队或系统会受到影响？"
```

**PM 需要提供**：
- **PM 信息**：姓名、职责、联系方式
- **技术负责人**：姓名、职责、联系方式
- **业务方**：部门、联系人、职责
- **相关方**：受影响的团队/系统列表

**⚠️ 重要**：
- 干系人必须具体到人（不能只写"技术团队"）
- 联系方式必须填写（便于沟通协作）
- 职责要明确（责任边界清晰）

### 第 4 步：项目约束

**AI 引导问题**：

```
AI: "明确项目约束条件：

     1. 时间约束：期望什么时候启动？什么时候交付？
     2. 资源约束：有哪些已知的人力/预算/技术限制？
     3. 依赖条件：需要其他项目/系统先完成什么？"
```

**PM 需要提供**：
- **时间约束**：启动时间、交付时间（可以是范围）
- **资源约束**：人力限制、预算限制、技术限制
- **依赖条件**：外部依赖、系统依赖、数据依赖

### 第 5 步：PM 确认

**AI 引导**：

```
AI: "P0 项目基本信息已完善。请检查确认：

     - [ ] 项目目标已明确且合理
     - [ ] 干系人已确认（具体到人）
     - [ ] 约束条件已记录
     - [ ] 可以开始基线建立
     
     确认无误后，我们可以进入下一阶段：创建 A0 基线文档。"
```

## 🔍 AI 检查清单

在完成 P0 填写后，AI 必须检查：

- [ ] **项目目标是可衡量的**（不是"提升体验"这类模糊表述）
- [ ] **干系人具体到人**（不是"技术团队"这类泛指）
- [ ] **成功标准可验证**（有明确的数字或状态）
- [ ] **紧迫性已说明**（为什么现在做）
- [ ] **约束条件已明确**（时间、资源、依赖）
- [ ] **PM 已确认**（勾选确认清单）

## 🎨 A2UI 架构图触发规则（P0 阶段）

**当 PM 在 P0 阶段描述项目整体结构、模块划分时，AI 必须主动生成架构图。**

### 触发信号词

| PM 提到的内容 | AI 触发行为 |
|-------------|-----------|
| "这个项目包含几个子系统..." | ✅ 主动生成系统架构图 |
| "涉及到前端、后端、数据库..." | ✅ 主动生成技术架构图 |
| "干系人之间的关系是..." | ✅ 主动生成干系人关系图 |
| "项目分为几个阶段..." | ✅ 主动生成项目阶段图 |

### AI 执行流程

```
1. PM 描述项目结构/干系人关系
   ↓
2. AI 识别到结构描述
   ↓
3. AI 主动提议：
   "我注意到您在描述项目结构。
    让我生成一个可视化的架构图帮助理解。"
   ↓
4. AI 生成 Diagram JSON 并写入 `.a2ui/current.json`
   ↓
5. AI 提示刷新浏览器查看
```

### 架构图组件

使用 `Diagram`, `Layer`, `DiagramGroup`, `Box`, `Arrow` 组件。
详见：`prd-a2ui-guide.md`

### 架构图保存规则

**当 PM 确认架构图后，AI 必须正式保存：**

- **保存位置**：`00_项目总览/P0_架构图/`
- **生成文件**：`.json` + `.html`
- **嵌入文档**：在 P0 文档中添加链接

## ✅ 完成标志

当以下条件全部满足时，P0 才算完成：

1. ✅ 所有章节都已填写（没有空白的 `<!-- 注释 -->`）
2. ✅ 成功标准是可衡量的（有具体指标）
3. ✅ 干系人信息完整（姓名、联系方式、职责）
4. ✅ PM 已勾选确认清单并签字
5. ✅ AI 检查清单全部通过

**完成后的下一步**：

```bash
prd baseline create A0  # 开始创建基线文档
```

## 🚨 常见错误与纠正

### 错误 1：目标不可衡量

❌ **错误**：
```markdown
**成功标准**:
- 用户体验提升
- 系统更稳定
```

✅ **正确**：
```markdown
**成功标准**:
- 用户满意度从 3.5 分提升至 4.5 分（满分 5 分）
- 系统可用性从 95% 提升至 99.5%
```

### 错误 2：干系人不具体

❌ **错误**：
```markdown
**技术负责人**:
- 姓名: 技术团队
```

✅ **正确**：
```markdown
**技术负责人**:
- 姓名: 张三
- 职责: 技术可行性把关、架构设计审批
- 联系方式: zhangsan@company.com / 13800138000
```

### 错误 3：约束条件过于模糊

❌ **错误**：
```markdown
**期望交付时间**: 尽快
```

✅ **正确**：
```markdown
**期望交付时间**: 2025 年 Q1（1-3 月），最晚不超过 3 月 31 日
```

## 📋 总结

P0_项目基本信息是整个 PRD 体系的**第一步**，必须认真填写。

### 核心要点

1. **P0 必须在 A0 之前完成**
2. **目标要可衡量**（不能是模糊表述）
3. **干系人要具体**（具体到人和联系方式）
4. **约束要明确**（时间、资源、依赖）
5. **PM 必须确认**（勾选清单并签字）

### AI 的角色

- 引导 PM 填写（不是替 PM 填写）
- 检查填写质量（目标是否可衡量、干系人是否具体）
- 分段写入和确认（不一次性生成）
- 确保完成度（所有章节都填写完整）

---

**记住：P0 是起点，不能跳过！**
