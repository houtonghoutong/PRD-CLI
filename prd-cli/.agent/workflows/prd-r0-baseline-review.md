# R0 基线审视工作流

本工作流用于在创建 A0/A1/A2 后，执行基线审视，为后续迭代规划提供稳定的起点。

---

## 🎯 R0 审视的本质：建立共识事实

R0 不是评价产品好坏，而是**客观审视现状，建立共识的基线**。

### 核心思维框架：

```
Step 1: 系统性审视 → 从头到尾走一遍系统
Step 2: 结构化梳理 → 用 MECE 原则梳理问题和机会
Step 3: 基线判定 → 判断是否适合作为稳定起点
```

---

## 📋 R0 审视的触发时机

### 必须满足的前置条件：

- ✅ A0（产品基础与范围说明）已完成
- ✅ A1（已上线功能清单）已完成
- ✅ A2（存量反馈汇总）已完成

**命令**：`prd baseline create R0`

---

## 🔍 R0 审视的流程

### Step 1: 系统性审视

**AI 的第一个任务（必须做）**：

```
AI: "在开始 R0 审视之前，我需要做系统性审视。
     
     让我从头到尾走一遍系统，基于 A0/A1/A2：
     
     1. 主要用户路径梳理
        - 根据 A1，识别核心用户路径
        - 每条路径的关键节点
     
     2. 实际使用情况
        - 用户在每个节点的实际体验
        - 是否有断点或痛点
     
     3. 与 A2 反馈的对应
        - 用户反馈的问题在哪个路径节点出现？
        - 数据异常对应哪个功能？
     
     请允许我基于 A0/A1/A2 完成这个梳理。"
```

**AI 必须输出：用户路径审视矩阵**

| 路径 | 关键节点 | 当前状态 | A2 反馈 | 问题等级 |
|-----|---------|---------|--------|---------|
| 创建 BP | 进入流程 | 正常 | 无 | ✅ |
| 创建 BP | 填写信息 | 有痛点 | 反馈 #3、#7 | ⚠️ |
| 创建 BP | 提交确认 | 缺失功能 | 反馈 #12 | ❌ |

---

### Step 2: 结构化梳理（MECE 原则）

**AI 的第二个任务：用 MECE 原则分类问题和机会**

#### 2.1 问题分类（ME - 相互独立）

```
AI: "基于系统审视，我用 MECE 原则分类问题：
     
     问题分类树：
     
     所有问题
     ├── 功能层面
     │   ├── 功能缺失（如：缺少进度指示）
     │   └── 功能不完善（如：保存不稳定）
     ├── 体验层面
     │   ├── 流程繁琐（如：步骤过多）
     │   └── 提示不清（如：错误提示模糊）
     └── 技术层面
         ├── 性能问题（如：加载慢）
         └── 稳定性问题（如：偶尔崩溃）
     
     检查：这些分类是否相互独立且完全穷尽？"
```

#### 2.2 优先级判断（CSF 识别）

```
AI: "识别关键问题（CSF）：
     
     哪些问题如果不解决，会严重影响后续规划？
     
     | 问题 | 影响范围 | 紧迫性 | 是否 CSF |
     |-----|---------|-------|---------|
     | 缺少进度指示 | 40% 用户放弃 | 高 | ✅ CSF |
     | 保存不稳定 | 30% 数据丢失 | 高 | ✅ CSF |
     | 加载慢 | 用户抱怨 | 中 | ❌ 非 CSF |
     
     建议：CSF 问题应在第一轮迭代解决。"
```

---

### Step 3: 机会点识别

**AI 的第三个任务：识别改进机会**

```
AI: "基于审视，识别改进机会：
     
     机会分类（MECE）：
     
     改进机会
     ├── 快速见效（低成本高收益）
     │   ├── 增加进度提示 ← 推荐优先
     │   └── 优化错误提示
     ├── 中长期优化（高成本高收益）
     │   ├── 流程重构
     │   └── 智能填充
     └── 创新尝试（高风险高收益）
         └── AI 辅助创建
     
     建议：第一轮迭代聚焦'快速见效'类机会。"
```

---

### Step 4: 风险与隐患

**AI 的第四个任务：风险评估**

```
AI: "风险评估（分类）：
     
     已知风险
     ├── 技术风险
     │   ├── 架构老旧，扩展性差 ← 高风险
     │   └── 依赖第三方服务不稳定
     └── 业务风险
         ├── 用户流失率上升 ← 高风险
         └── 竞品压力
     
     高风险项：
     1. 架构老旧 - 建议：评估重构成本
     2. 用户流失 - 建议：第一轮迭代必须解决 CSF 问题"
```

---

### Step 5: 基线稳定性判定

**AI 的最终判断（必须明确）**：

```
AI: "基线稳定性评估：
     
     判断标准：
     1. 核心功能是否完整？
        - 根据 A1，核心功能：✅ 完整
     
     2. 是否存在阻断性问题？
        - 阻断性问题：❌ 存在（保存不稳定）
     
     3. 是否适合作为稳定基线？
        - 需要先解决保存稳定性问题
     
     结论：
     - [ ] ✅ 可以作为稳定基线（无阻断性问题）
     - [x] ⚠️ 需要先解决关键问题（保存稳定性）
     - [ ] ❌ 不适合作为基线（重大缺陷过多）
     
     建议：
     1. 先修复保存稳定性（阻断性问题）
     2. 修复后即可作为基线
     3. 其他问题可在迭代中逐步解决"
```

```
问题

```

```
|-----|-----|-------|-----|

```

```
机会
├── 快速见效
```

## 💬 AI 与 PM 的对话方式

### 错误方式（走过场）：

```
AI: "我看了 A0/A1/A2，产品还不错，可以作为基线。"
← 没有审视，没有分析
```

### 正确方式（系统审视）：

```
AI: "我刚刚系统性地审视了产品基线（A0/A1/A2）。
     
     让我分享审视结果：
     
     ## 用户路径审视
     我识别了 3 条核心路径：[列出路径]
     
     ## 主要发现
     问题分类（MECE）：
     - 功能层面：2 个问题
     - 体验层面：5 个问题
     - 技术层面：1 个问题
     
     ## CSF 识别
     关键问题（必须解决）：
     1. 保存稳定性 ← 阻断性问题
     2. 进度提示缺失 ← 高影响
     
     ## 建议
     建议先解决'保存稳定性'，之后可作为稳定基线。
     
     您同意这个判断吗？"

PM: [确认或补充]

AI: "好的，我现在将完整的审视结果写入 R0 报告。"
```

## ⚠️ AI 行为红线

### AI 绝对禁止：

1. ❌ 不做系统审视就给结论
2. ❌ 笼统地说"还不错"
3. ❌ 不用 MECE 分类问题
4. ❌ 不识别 CSF
5. ❌ 基线判定不明确（模棱两可）

### AI 必须做到：

1. ✅ 基于 A0/A1/A2 系统审视
2. ✅ 用 MECE 原则分类问题
3. ✅ 识别 CSF（关键问题）
4. ✅ 构建用户路径审视矩阵
5. ✅ 给出明确的基线判定

## 🔄 后续流程

### R0 完成后：

```
AI: "R0 基线审视已完成。
     
     结论：[稳定基线 / 需先解决问题 / 不适合]
     
     下一步：
     1. 如果是'稳定基线' → 可以开始第一轮迭代
        运行：prd iteration new
     
     2. 如果是'需先解决问题' → 先解决阻断性问题
        - 在当前系统修复
        - 修复后更新 A1
        - 重新执行 R0
     
     3. 如果是'不适合' → 产品可能需要重构
        - 评估重构成本和收益
        - 决定是否值得投入
     
     请问您要如何处理？"
```
