# R0 基线审视工作流

本工作流用于在创建 A0/A1/A2 后，执行基线审视，为后续迭代规划提供稳定的起点。

---

## 🛡️ 基线准备（从 prd-baseline-check.md 合并）

### 铁律：先有基线，再谈规划

**AI 在开始 B1 讨论之前，必须检查 A0/A1/A2 是否存在！**

### 用户提供信息时的归档规则

**当 PM 提供"已有功能说明"时：**

```
PM: "这是我们系统的已有功能清单..."

✅ 正确做法：
AI: "感谢！这份内容应该归档为 A1_已上线功能清单.md。
     我现在帮你创建 A1 文档并写入，可以吗？"
PM: "可以"
AI: [执行 prd baseline create A1]
AI: [将用户提供的内容写入 A1]
AI: "A1 已创建并归档！"

❌ 错误做法：
AI: "好的，那我们来讨论 B1 规划吧..." 
# 跳过了归档步骤！
```

**当 PM 提供"用户反馈"或"问题描述"时：**

```
AI: "这些反馈应该归档为 A2_存量反馈汇总.md，可以吗？"
[获得确认后创建 A2 并写入]
```

### 用户信息归档对照表

| PM 提供的信息 | 应归档到 |
|--------------|---------|
| 产品介绍、定位、用户群 | A0 |
| 已有功能列表、功能说明 | A1 |
| 用户反馈、问题描述、数据异常 | A2 |
| 能力边界、技术限制 | A0 |

### AI 归档禁令

- ❌ 在对话中讨论了功能，但没有归档到 A1
- ❌ 在对话中讨论了问题，但没有归档到 A2
- ❌ 在 A 类文档不完整的情况下开始 B 类讨论
- ❌ 只是"记住"用户说的，而不正式写入文档

## 🎯 R0 审视的本质：建立共识事实

R0 不是评价产品好坏，而是**客观审视现状，建立共识的基线**。

### 核心思维框架：

```
Step 1: 系统性审视 → 从头到尾走一遍系统
Step 2: 结构化梳理 → 用 MECE 原则梳理问题和机会
Step 3: 基线判定 → 判断是否适合作为稳定起点
```

## 🛡️ 职责划分（PM 判定，AI 执行）

### PM 职责（必须由人完成）

**PM 必须亲自做的（不可替代）**：

1. **确认审视范围**
   ```
   PM: "我确认审视范围包括：
        - 核心模块：创建 BP、编辑 BP、查看 BP
        - 不包括：审批流程（尚未上线）
        
        请基于此范围进行审视。"
   ```

2. **确认哪些现象/问题为"事实成立"**
   ```
   - 避免把主观感受当事实
   - AI 提出的问题，PM 需确认是否真实存在
   
   AI: "我发现'创建过程可能让用户放弃'这个问题"
   PM: "确认，这是基于 A2 反馈 #3、#7 的事实" ✅
   
   AI: "我觉得界面不够美观"
   PM: "这是主观感受，不纳入" ❌
   ```

3. **给出基线审视结论是否采纳**
   ```
   AI: "我的审视结论是：需要先解决保存稳定性问题"
   
   PM: [决策]
   - ✅ 采纳全部结论
   - ⚠️ 部分采纳（说明采纳范围）
   - ❌ 不采纳（说明理由）
   ```

### AI 职责（自动执行）

**AI 必须做的（不需 PM 干预）**：

1. **基于 A0/A1/A2 组织"从头到尾的用户路径审视"**
   - 自动梳理所有用户路径
   - 识别关键节点
   - 标注痛点位置

2. **归纳问题与机会清单（分类/聚类/去重）**
   - 用 MECE 原则分类问题
   - 去除重复问题
   - 聚类相似问题

3. **标注风险与影响面（但不做价值取舍结论）**
   ```
   AI: "问题：缺少进度指示
        影响面：40% 用户（基于 A2 #3、#7、#12）
        风险等级：🔴 高"
   
   ❌ AI 不应该说："所以必须做"
   ✅ AI 应该说："请 PM 决定是否解决"
   ```

4. **输出结构化报告与可追溯引用点（指向 A 类条目）**
   - 每个问题必须引用 A1/A2 的具体条目
   - 报告必须结构化（MECE 分类）

   ```
        - 在当前系统修复问题
        - 修复后更新 A1
   ```

2. ❌ **R0 完成后，禁止主动提及"R1"**
   ```
   错误示例：
   AI: "如果要基于此规划，请允许我进入 R1 启动条件检查"
   ← 容易误导！不应该在 R0 阶段提及 R1
   
   正确做法：
   AI: "下一步请创建第一轮迭代：prd iteration new"
   ← 只说明紧接着的下一步，不要跳跃
   ```

3. ❌ **不得替 PM 判定"必须做/不做"**
   ```
   错误示例：
   AI: "进度指示器必须做，保存功能可以延后"
   ← 这是价值取舍，AI 不能做

   AI: "进度指示器：影响 40% 用户，风险高
        保存功能：影响 30% 用户，风险高
        请 PM 判定优先级"
   ```

## ⚠️ 重要：两种审视模式

### 模式 1：基于用户反馈的审视（A2 有内容）

- 以 A2 中的用户反馈为依据
- 验证反馈的真实性和影响范围
- 基于事实数据进行审视

### 模式 2：AI 主动审视（A2 为空或不足）

**当 A2 为空或反馈不足时，AI 必须主动审视！**

**AI 扮演角色：挑剔的用户**

```
AI: "虽然 A2 中没有用户反馈，但我仍需做极致审视。
     
     我现在扮演一个挑剔的用户，从最严格的标准审视产品。
     
     基于 A1 的功能清单，我会：
     1. 模拟用户使用每个功能
     2. 找出所有可能的痛点和问题
     3. 提出改进建议
     
     即使用户没有反馈，我也要主动发现问题。"
```

**AI 必须从以下角度挑刺：**

1. **新手用户视角**
   - 第一次使用能找到功能吗？
   - 操作步骤是否清晰？
   - 有没有"不知道怎么用"的地方？

2. **高频用户视角**
   - 重复操作是否繁琐？
   - 能否提高效率？
   - 有没有快捷方式？

3. **极端场景**
   - 网络断开怎么办？
   - 数据异常怎么办？
   - 并发操作会出问题吗？

4. **竞品对比**
   - 同类产品做得更好的地方？
   - 我们的明显短板是什么？

5. **潜在风险**
   - 哪些功能可能出错？
   - 用户可能遇到什么困惑？
   - 技术债务在哪里？

## 🔍 R0 审视的流程

### Step 1: 系统性审视

#### 如果 A2 有用户反馈：

**AI 的第一个任务（必须做）**：

```
AI: "在开始 R0 审视之前，我需要做系统性审视。
     
     让我从头到尾走一遍系统，基于 A0/A1/A2：
     
     1. 主要用户路径梳理
        - 根据 A1，识别核心用户路径
        - 每条路径的关键节点
     
     2. 实际使用情况
        - 用户在每个节点的实际体验
        - 是否有断点或痛点
     
     3. 与 A2 反馈的对应
        - 用户反馈的问题在哪个路径节点出现？
        - 数据异常对应哪个功能？
     
     请允许我基于 A0/A1/A2 完成这个梳理。"
```

#### 如果 A2 为空或不足（AI 主动挑刺模式）：

**AI 必须主动扮演挑剔用户：**

```
AI: "我注意到 A2 中没有用户反馈（或反馈不足）。
     
     但这不代表产品没有问题，可能是：
     - 用户还没来得及反馈
     - 用户默默忍受问题
     - 我们没有收集反馈的渠道
     
     所以我现在扮演一个极度挑剔的用户，主动审视产品。
     
     基于 A1 的功能清单，让我逐个'挑刺'："
```

**AI 主动挑刺流程：**

##### 1. 逐功能深度拷问

```
AI: "功能 1.1：创建 BP
     
     【新手用户视角】
     - ❓ 第一次进入，用户知道如何开始吗？
     - ❓ 创建按钮是否明显？
     - ❓ 有引导提示吗？
     
     【操作流程审视】
     - ❓ 需要几步才能完成创建？
     - ❓ 每一步的目的是否清晰？
     - ❓ 是否有不必要的步骤？
     
     【极端场景】
     - ❓ 填到一半退出，数据会丢失吗？
     - ❓ 网络断开时会怎样？
     - ❓ 必填项未填时提示清楚吗？
     
     【潜在问题（我的挑剔意见）】
     1. ⚠️ 缺少进度指示 → 用户不知道填到哪一步
     2. ⚠️ 没有自动保存 → 退出会丢失数据
     3. ⚠️ 错误提示不清晰 → 用户不知道哪里填错了
     4. ⚠️ 步骤过多（12步）→ 用户可能中途放弃
     
     【严重程度评估】
     - 🔴 严重（缺少进度指示）
     - 🟠 重要（没有自动保存）
     - 🟡 一般（错误提示）
```

##### 2. 用户体验挑刺

```
AI: "【整体体验审视】
     
     作为挑剔用户，我的感受：
     
     ❌ 痛点 1：创建过程太长
        - 需要填写 12 个步骤
        - 没有进度提示，不知道还要填多久
        - 预计用户放弃率会很高
     
     ❌ 痛点 2：没有容错机制
        - 填错数据没有提示
        - 提交失败后数据丢失
        - 用户会很沮丧
     
     ❌ 痛点 3：缺少帮助
        - 某些字段不知道怎么填
        - 没有示例或帮助文档
        - 新手会很困惑
     
     💡 如果我是用户：
        '这个系统好难用，填个东西要这么久，
         而且填到一半还可能丢失，太糟糕了！'"
```

##### 3. 竞品对比挑刺

```
AI: "【竞品对比】
     
     我对比了同类产品（如飞书目标、Teambition）：
     
     我们落后的地方：
     - ❌ 飞书有进度条，我们没有
     - ❌ Teambition 有自动保存，我们没有
     - ❌ 飞书有智能提示，我们没有
     
     我们的优势：
     - ✅ 流程定制化（但也导致复杂）
     
     结论：在基础体验上，我们明显落后。"
```

##### 4. 技术债务挑刺

```
AI: "【技术层面审视】
     
     作为挑剔的技术用户，我的观察：
     
     ⚠️ 性能问题：
        - 加载时间 > 3 秒（用户容忍度 < 2 秒）
        - 提交响应慢（可能丢失用户耐心）
     
     ⚠️ 稳定性问题：
        - 没有看到错误边界处理
        - 并发编辑可能出问题
     
     ⚠️ 兼容性问题：
        - 移动端体验差
        - 某些浏览器可能不兼容"
```

**AI 必须输出：主动审视矩阵**

| 功能 | 新手痛点 | 高频痛点 | 极端场景 | 竞品差距 | 严重度 |
|-----|---------|---------|---------|---------|-------|
| 创建 BP | 不知道怎么开始 | 步骤繁琐 | 数据丢失 | 无进度条 | 🔴 |
| 编辑 BP | - | 保存慢 | 并发冲突 | - | 🟠 |
| ... | ... | ... | ... | ... | ... |

---

**AI 必须输出：用户路径审视矩阵**

| 路径 | 关键节点 | 当前状态 | A2 反馈 | 问题等级 |
|-----|---------|---------|--------|---------|
| 创建 BP | 进入流程 | 正常 | 无 | ✅ |

### Step 2: 结构化梳理（MECE 原则）

**AI 的第二个任务：用 MECE 原则分类问题和机会**

#### 2.1 问题分类（ME - 相互独立）

```
AI: "基于系统审视，我用 MECE 原则分类问题：
     
     问题分类树：
     
     所有问题
     ├── 功能层面
     │   ├── 功能缺失（如：缺少进度指示）
     │   └── 功能不完善（如：保存不稳定）
     ├── 体验层面
     │   ├── 流程繁琐（如：步骤过多）
     │   └── 提示不清（如：错误提示模糊）
     └── 技术层面
         ├── 性能问题（如：加载慢）
         └── 稳定性问题（如：偶尔崩溃）
     
     检查：这些分类是否相互独立且完全穷尽？"
```

#### 2.2 优先级判断（CSF 识别）

```
AI: "识别关键问题（CSF）：
     
     哪些问题如果不解决，会严重影响后续规划？
     
     | 问题 | 影响范围 | 紧迫性 | 是否 CSF |
     |-----|---------|-------|---------|
     | 缺少进度指示 | 40% 用户放弃 | 高 | ✅ CSF |
     | 保存不稳定 | 30% 数据丢失 | 高 | ✅ CSF |
     | 加载慢 | 用户抱怨 | 中 | ❌ 非 CSF |
     
     建议：CSF 问题应在第一轮迭代解决。"
```

```
     1. 架构老旧 - 建议：评估重构成本
     2. 用户流失 - 建议：第一轮迭代必须解决 CSF 问题"
```

### Step 5: 基线稳定性判定

**AI 的最终判断（必须明确）**：

```
AI: "基线稳定性评估：
     
     判断标准：
     1. 核心功能是否完整？
        - 根据 A1，核心功能：✅ 完整
     
     2. 是否存在阻断性问题？
        - 阻断性问题：❌ 存在（保存不稳定）
     
     3. 是否适合作为稳定基线？

     结论：
     - [ ] ✅ 可以作为稳定基线（无阻断性问题）
     - [x] ⚠️ 需要先解决关键问题（保存稳定性）
     - [ ] ❌ 不适合作为基线（重大缺陷过多）
     
     建议：
     1. 先修复保存稳定性（阻断性问题）
     2. 修复后即可作为基线
     3. 其他问题可在迭代中逐步解决"
```

## 🔄 R0 完成后的正确流程

### ⚠️ 重要：R0 是一个独立的审视阶段

**R0 完成后，AI 的职责结束，不得继续任何操作！**

### R0 完成时的标准话术

```
AI: "✅ R0 基线审视已完成。
     
     📊 审视结论：[稳定基线 / 需先解决问题 / 不适合]
     
     ⚠️ 我的职责到此为止，不会自动执行任何后续操作。
     
     下一步完全由您决定：
     
     ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
     
     【情况 1】如果审视结论是"稳定基线"：
     
     您可以开始第一轮迭代，请按以下顺序操作：
     
     第 1 步：创建第一轮迭代
     命令：prd iteration new
     （这会自动生成 R1_规划启动条件检查.md）
     
     第 2 步：填写并确认 R1 启动条件
     打开 R1_规划启动条件检查.md，确认 3 个条件：
     1. 问题真实存在（在 A1/A2 中有依据）
     2. 值得单独规划（不是小修小补）
     3. 问题已理解清楚（边界明确）
     ⚠️ 任何一条不满足，不要继续！
     
     第 3 步：创建 B1 需求规划草案
     命令：prd plan create B1
     然后与我对话填写 B1
     
     第 4 步：创建 B2 规划拆解
     命令：prd plan create B2
     然后与我对话填写 B2
     
     第 5 步：请求 R1 规划审视
     命令：prd review r1
     （这会生成 R1_规划审视报告.md）
     我将审视 B1/B2 是否有资格冻结为 B3
     
     ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
     
     【情况 2】如果审视结论是"需先解决问题"：
     
     请先修复关键问题：
     1. 在当前系统中修复问题
     2. 修复后更新 A1_已上线功能与流程清单.md
     3. 重新执行 R0：prd baseline create R0
     
     ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
     
     【情况 3】如果审视结论是"不适合作为基线"：
     
     产品可能需要重构，请评估：
     - 重构成本和收益
     - 是否值得投入
     - 还是考虑重新设计
     
     ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
     
     请问您要如何处理？"
```
