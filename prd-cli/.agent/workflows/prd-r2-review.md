# R2 版本审视工作流

## 适用场景
当完成 C0、C1 文档后，准备冻结为 C3 之前，必须执行 R2 审视。

## 前置条件
- [ ] B3 规划冻结归档已完成
- [ ] C0 版本范围声明已完成
- [ ] C1 版本需求清单已完成

## 执行步骤

### 1. 启动 R2 审视
```bash
prd review r2
```

### 2. AI 执行审视指令

你现在扮演【版本一致性审视官】。

我将提供以下输入:
- B3(已冻结的规划文档)
- C0 / C1(版本范围与版本需求清单)

你的任务不是评判方向，而是判断:
【该版本是否忠实执行了既定规划】

请严格按以下 5 个维度进行审视，并逐条给出判断依据:

#### 维度一：版本目标一致性
- C0 的版本目标是否能在 B3 中找到明确对应
- 是否存在规划外目标

#### 维度二：版本范围偏移检查
- 版本包含/不包含是否符合 B3 边界
- 是否存在隐性扩展或"顺手加戏"

#### 维度三：规划覆盖完整性
- B3 中的核心规划点是否在 C1 中得到体现
- 是否存在规划被版本拆没的情况

#### 维度四：需求粒度成熟度
- 每条需求是否达到版本级、可理解、可评估
- 是否仍停留在规划语言

#### 维度五：进入执行准备度
- 是否已不再需要产品侧做判断决策
- 是否只剩实现与执行问题

### 3. 给出最终结论

最后，请给出唯一结论(三选一):
- 【通过】允许版本冻结
- 【有条件通过】需修订的具体需求项
- 【不通过】禁止版本冻结

### 4. 填写审视报告

在生成的 `R2_版本审视报告.md` 中填写:
1. 每个维度的审视结果
2. 具体的判断依据
3. 最终结论

### 5. 根据结论决定下一步

**如果通过**:
```bash
prd version freeze  # 生成 C3 版本冻结归档
```

**如果有条件通过**:
1. 修订指定的需求项
2. 更新 C0/C1 文档
3. 重新执行 R2 审视

**如果不通过**:
1. 分析版本与规划的偏差
2. 调整版本范围或需求
3. 必要时回到 B3 重新审视规划

## ⚠️ 重要约束

- 禁止讨论规划是否正确
- 禁止提出新增需求
- 只判断版本是否忠实执行了既定规划
- 必须给出明确的通过/不通过结论
- 不通过时必须说明具体原因

## 审视重点

R2 与 R1 的区别:
- **R1**: 审视规划本身是否合理
- **R2**: 审视版本是否符合规划

R2 的核心任务:
- 确保 C 忠于 B
- 防止版本执行时"走样"
- 阻止"顺手加戏"
