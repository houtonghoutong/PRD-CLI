# R1 规划审视工作流

## 适用场景
当完成 B1、B2 文档后，准备冻结为 B3 之前，必须执行 R1 审视。

## 前置条件
- [ ] A 类基线文档已完成 (A0, A1, A2)
- [ ] B1 需求规划草案已完成
- [ ] B2 规划拆解与范围界定已完成

## 执行步骤

### 1. 启动 R1 审视
```bash
prd review r1
```

### 2. AI 执行审视指令

你现在扮演【产品规划审视官】。

我将提供以下输入:
- A 类文档(产品现状与基线)
- B1 / B2(需求规划草案与拆解)

你的任务不是提出新方案，而是判断:
【这份规划是否有资格被冻结为 B3】

请严格按以下 5 个维度进行审视，并逐条给出判断依据:

#### 维度一：目标清晰性
- 是否能用一句话说明本次规划要解决的核心问题
- 是否存在多个不相关目标混杂

#### 维度二：场景真实性
- 使用场景是否基于 A 类现状
- 是否依赖当前不存在的能力

#### 维度三：现状一致性
- 是否明确区分"已有能力改进"与"新增能力"
- 是否存在从零重建的倾向

#### 维度四：范围收敛性
- 是否明确说明"不做什么"
- 是否存在明显范围膨胀风险

#### 维度五：版本化准备度
- 是否已不再讨论"值不值得做"
- 是否具备进入版本拆分的条件

### 3. 给出最终结论

最后，请给出唯一结论(三选一):
- 【通过】可冻结为 B3
- 【有条件通过】需补齐的明确项
- 【不通过】不可进入 B3

### 4. 填写审视报告

在生成的 `R1_规划审视报告.md` 中填写:
1. 每个维度的审视结果
2. 具体的判断依据
3. 最终结论

### 5. 根据结论决定下一步

**如果通过**:
```bash
prd plan freeze  # 生成 B3 规划冻结归档
```

**如果有条件通过**:
1. 补充缺失内容
2. 更新 B1/B2 文档
3. 重新执行 R1 审视

**如果不通过**:
1. 分析不通过原因
2. 重新梳理规划
3. 必要时回到 R1 启动条件检查

## ⚠️ 重要约束

- 禁止输出任何新的需求或设计建议
- 只审视现有规划是否满足冻结条件
- 必须给出明确的通过/不通过结论
- 不通过时必须说明具体原因
